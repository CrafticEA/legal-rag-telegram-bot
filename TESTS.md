# TESTS.md

## 1. Цель тестирования

Целью тестирования является проверка корректности работы RAG-сервиса с технической точки зрения, а также оценка удобства и полезности системы для конечных пользователей (юристов).

Тестирование разделено на:
- техническое 
- пользовательское 

---

## 2. Техническое тестирование

### 2.1 Проверка API

Проверялись основные эндпоинты:

#### POST /build
Ожидаемое поведение:
- успешная обработка документов
- построение индекса
- возврат статистики

Результат:
- индекс успешно создается
- ошибки возникают только при:
  - отсутствии текста в документе
  - неверных doc_id
  - отсутствии файлов

#### POST /retrieve
Ожидаемое поведение:
- возврат релевантных чанков

Результат:
- корректно возвращаются top-k чанков
- источники и страницы отображаются правильно

#### GET /health
Результат:
- стабильно возвращает статус сервиса

---

### 2.2 Тестирование пайплайна RAG

Проверялись этапы:

#### Парсинг документов
- PDF: текст извлекается, но возможны потери форматирования
- DOCX: работает стабильно
- TXT: без проблем

Проблемы:
- сложные PDF (сканы) не обрабатываются

---

#### Чанкинг
Результат:
- чанки формируются корректно
- overlap работает
- нет обрывов текста посередине слов

Проблемы:
- иногда слишком мелкие чанки на коротких абзацах

---

#### Эмбеддинги
Результат:
- модель стабильно работает
- скорость зависит от CPU/GPU

Проблемы:
- на CPU медленно при больших объемах документов

---

#### Индексация (FAISS)
Результат:
- индекс создается корректно
- поиск быстрый

Проблемы:
- отсутствует персистентное обновление (индекс пересобирается полностью)

---

#### Retrieval + Reranking
Результат:
- релевантные чанки чаще всего попадают в top-k
- reranking улучшает качество

Проблемы:
- иногда нерелевантные куски попадают в выдачу при слабом запросе

---

### 2.3 Нагрузочное тестирование

Условия:
- 10–20 документов
- до ~5000 чанков

Результат:
- build: несколько секунд (CPU)
- retrieve: < 1 секунды

Проблемы:
- при росте объема данных нет оптимизации (нет IVF, HNSW)

---

## 3. Пользовательское тестирование

### 3.1 Состав фокус-группы

Фокус-группа: 3–5 человек

Состав:
- студенты-юристы
- начинающие юристы
- технический пользователь (разработчик)

---

### 3.2 Сценарий тестирования

Пользователям было предложено:

1. Загрузить документы (договоры, кейсы)
2. Задать вопросы:
   - "Какие риски есть в этом договоре?"
   - "Какие обязанности у стороны?"
   - "Есть ли штрафные санкции?"

3. Оценить:
   - релевантность ответов
   - удобство
   - понятность

---

### 3.3 Результаты

#### Плюсы

- Быстрый поиск по документам
- Ответы опираются на текст (нет полной "галлюцинации")
- Удобно для первичного анализа документов
- Экономит время при работе с большими файлами

---

#### Минусы

- Иногда ответы слишком общие
- Нет полноценной "юридической логики", только извлечение
- Требуется хорошая формулировка запроса
- Нет объяснения, почему выбран именно этот ответ



## 4. Выводы

1. Система технически работоспособна:
   - корректно строит индекс
   - выполняет поиск
   - возвращает релевантные фрагменты

2. Основная ценность:
   - ускорение анализа документов
   - снижение ручного чтения

3. Ограничения:
   - нет полноценного reasoning
   - качество зависит от retrieval
   - чувствительность к формулировке запроса

---

## 5. Рекомендации по улучшению

### Технические
- добавить OCR для сканов
- перейти на более сложные индексы (IVF, HNSW)
- оптимизировать работу на CPU

### Продуктовые
- добавить генерацию ответа через LLM поверх retrieval
- объяснение источников (explainability)
- улучшить UX Telegram-бота

- решает задачу поиска по юридическим документам
